 fsmon 是什么

 ```
   fsmon 是一个基于 Linux fanotify FID 模式的文件系统实时监控守护进程
   能监控文件变化，并追溯到引起变化的进程（含完整进程链）
 ```

 架构：

 ```
   ┌─────────────────────────────────────────────────────────────┐
   │                        fsmon daemon                         │
   │  fanotify (FID) ──┐                                        │
   │  proc-connector ──┤──▶ tokio 事件循环 ──▶ 日志写入          │
   │  inotify ─────────┘          │                              │
   │                              ▼                              │
   │                     Unix Socket 广播                        │
   └─────────────────────────────────────────────────────────────┘
 ```

 ────────────────────────────────────────────────────────────────────────────────

 fsmon 的性能关键路径

 ┌───────────────────┬───────────────────────────────────────┬───────────────────────────────┐
 │ 热点              │ 具体操作                              │ 为什么是热点                  │
 ├───────────────────┼───────────────────────────────────────┼───────────────────────────────┤
 │ fanotify 事件读取 │ read_fid_events_cached()              │ 每个文件事件都走这里          │
 ├───────────────────┼───────────────────────────────────────┼───────────────────────────────┤
 │ 路径解析          │ 三级优先链：本地 map → 缓存 → syscall │ rm -rf 场景下大量 handle 解析 │
 ├───────────────────┼───────────────────────────────────────┼───────────────────────────────┤
 │ 进程树查询        │ proc_tree::is_descendant()            │ 每个事件都要判断是否匹配进程  │
 ├───────────────────┼───────────────────────────────────────┼───────────────────────────────┤
 │ 事件过滤          │ should_output()                       │ 每个事件都要过一遍            │
 ├───────────────────┼───────────────────────────────────────┼───────────────────────────────┤
 │ 日志写入          │ BufWriter + fdatasync                 │ 磁盘 I/O 瓶颈                 │
 ├───────────────────┼───────────────────────────────────────┼───────────────────────────────┤
 │ 缓存操作          │ moka W-TinyLFU                        │ 高并发读写                    │
 └───────────────────┴───────────────────────────────────────┴───────────────────────────────┘

 ────────────────────────────────────────────────────────────────────────────────

 用 Perf 做性能测试的完整流程

 ### 第一步：编译带调试信息的版本

 ```bash
   # Cargo.toml 的 release profile 默认 strip=true，需要临时关掉
   # 或者用 debug 版本（但性能特征不同）

   # 方案 A：修改 Cargo.toml
   [profile.release]
   strip = false      # 不去掉符号表
   debug = true       # 包含 DWARF 调试信息

   # 方案 B：用 debug 版本测试（开销更高，但符号完整）
   cargo build

   # 方案 C：单独编译一个 profiling 版本
   [profile.profiling]
   inherits = "release"
   strip = false
   debug = true
   lto = true
 ```

 │ 为什么要这样做？ perf 需要符号表才能把地址翻译成函数名。strip=true 会去掉符号，perf report 就只能看到 [unknown]。

 ────────────────────────────────────────────────────────────────────────────────

 ### 第二步：启动 fsmon 并制造负载

 ```bash
   # 终端 1：启动 fsmon daemon
   ./target/release/fsmon daemon &

   # 终端 2：注册监控一个目录
   ./target/release/fsmon add _global --path /tmp/fsmon_bench -r -t all

   # 终端 3：制造文件操作负载
   # 方案 A：简单批量操作
   for i in $(seq 1 10000); do
       echo "test" > /tmp/fsmon_bench/file_$i.txt
   done

   # 方案 B：用 fsmon 自带的 benchmark
   cd benchmark && bash events_run.sh
 ```

 ────────────────────────────────────────────────────────────────────────────────

 ### 第三步：perf stat — 快速摸底

 ```bash
   # 获取 fsmon 的 PID
   PID=$(pidof fsmon)

   # 统计 30 秒内的硬件指标
   sudo perf stat -p $PID -- sleep 30
 ```

 输出示例：

 ```
    Performance counter stats for process 'fsmon' (pid 12345):

             30,000 msec task-clock                #    0.998 CPUs utilized
              1,234      context-switches          #   41.133 /sec
                 12      cpu-migrations            #    0.400 /sec
        2,345,678,901      cycles                  #    3.45 GHz
        1,234,567,890      instructions            #    0.53  insn per cycle
          123,456,789      cache-references        #    4.115 M/sec
            1,234,567      cache-misses            #    1.000% of all cache refs
           12,345,678      branch-misses           #    0.412 /sec
 ```

 关注什么：

 ┌──────────────────────┬──────────┬───────────────────────────────────┐
 │ 指标                 │ 理想值   │ 如果不理想说明什么                │
 ├──────────────────────┼──────────┼───────────────────────────────────┤
 │ IPC (insn per cycle) │ > 1.0    │ < 1.0 可能是 cache 问题或内存瓶颈 │
 ├──────────────────────┼──────────┼───────────────────────────────────┤
 │ cache-misses%        │ < 5%     │ 过高说明数据局部性差              │
 ├──────────────────────┼──────────┼───────────────────────────────────┤
 │ branch-misses        │ 越低越好 │ 可能有大量分支预测失败            │
 ├──────────────────────┼──────────┼───────────────────────────────────┤
 │ context-switches     │ 越低越好 │ 过高可能是锁竞争                  │
 └──────────────────────┴──────────┴───────────────────────────────────┘

 ────────────────────────────────────────────────────────────────────────────────

 ### 第四步：perf record — 采集调用栈

 ```bash
   # 采集 30 秒，99Hz 采样率，包含调用栈
   sudo perf record -F 99 -g -p $PID -- sleep 30

   # 同时制造负载（另一个终端）
   for i in $(seq 1 10000); do echo "test" > /tmp/fsmon_bench/file_$i.txt; done
 ```

 参数解释：

 ┌─────────────┬──────────────────────────────────────────┐
 │ 参数        │ 含义                                     │
 ├─────────────┼──────────────────────────────────────────┤
 │ -F 99       │ 每秒采样 99 次（避免与时钟对齐产生共振） │
 ├─────────────┼──────────────────────────────────────────┤
 │ -g          │ 记录调用栈（call graph）                 │
 ├─────────────┼──────────────────────────────────────────┤
 │ -p $PID     │ 只采样指定进程                           │
 ├─────────────┼──────────────────────────────────────────┤
 │ -- sleep 30 │ 采样 30 秒                               │
 └─────────────┴──────────────────────────────────────────┘

 ────────────────────────────────────────────────────────────────────────────────

 ### 第五步：perf report — 交互式分析

 ```bash
   sudo perf report
 ```

 界面：

 ```
   Overhead  Command  Shared Object      Symbol
     25.32%  fsmon    fsmon              [.] fid_parser::read_fid_events_cached
     18.45%  fsmon    libc.so.6          [.] __memcpy_avx2
     12.67%  fsmon    fsmon              [.] proc_tree::is_descendant
      8.91%  fsmon    fsmon              [.] moka::cache::get
      6.23%  fsmon    libc.so.6          [.] __epoll_wait
      ...
 ```

 操作：
 - 按 Enter 展开某个函数，看它的调用者/被调用者
 - 按 a 查看该行的汇编代码
 - 按 s 按符号排序
 - 按 d 只显示 dso（动态库）

 ────────────────────────────────────────────────────────────────────────────────

 ### 第六步：生成火焰图

 ```bash
   # 需要先安装 FlameGraph 工具
   git clone https://github.com/brendangregg/FlameGraph

   # 导出 perf 数据
   sudo perf script > fsmon.perf

   # 生成火焰图
   FlameGraph/stackcollapse-perf.pl fsmon.perf | \
       FlameGraph/flamegraph.pl > fsmon_flame.svg
 ```

 火焰图怎么看：

 ```
   宽度 = 占 CPU 时间的比例
   颜色 = 随机（无特殊含义）
   x 轴 = 调用栈（从底到顶）
   y 轴 = 调用深度

   找"宽平顶" → 那就是 CPU 时间花得最多的函数
 ```

 ────────────────────────────────────────────────────────────────────────────────

 ### 第七步：针对性分析

 #### 如果 read_fid_events_cached 很热：

 ```bash
   # 看具体是哪一行慢
   sudo perf annotate read_fid_events_cached
 ```

 可能发现：
 - resolve_file_handle 系统调用太多 → 缓存命中率低
 - handle_map 的 HashMap 操作 → 考虑用更快的 hash 函数

 #### 如果 memcpy 很热：

 ```bash
   # 看是谁在调用 memcpy
   sudo perf report --sort=dso,symbol --parent=read_fid_events_cached
 ```

 可能发现：
 - 事件结构体拷贝 → 改用引用或零拷贝
 - 路径字符串拷贝 → 用 Arc<str> 或 bytes::Bytes

 #### 如果 is_descendant 很热：

 ```bash
   # 看进程树查询的开销
   sudo perf stat -e 'cycles,instructions,cache-misses' -p $PID -- sleep 10
 ```

 可能发现：
 - 进程树太大 → 限制追踪深度
 - 查询太频繁 → 加缓存

 ────────────────────────────────────────────────────────────────────────────────

 完整命令速查表

 ```bash
   # 1. 编译带符号的版本
   cargo build --profile profiling

   # 2. 启动 fsmon
   ./target/profiling/fsmon daemon &
   fsmon add _global --path /tmp/bench -r -t all

   # 3. perf stat 快速摸底
   sudo perf stat -p $(pidof fsmon) -- sleep 30

   # 4. perf record 采集调用栈
   sudo perf record -F 99 -g -p $(pidof fsmon) -- sleep 30

   # 5. perf report 交互分析
   sudo perf report

   # 6. 生成火焰图
   sudo perf script > fsmon.perf
   FlameGraph/stackcollapse-perf.pl fsmon.perf | FlameGraph/flamegraph.pl > flame.svg

   # 7. perf annotate 看具体哪一行慢
   sudo perf annotate read_fid_events_cached
 ```

 针对 fsmon 的特殊注意事项

 ┌─────────────────┬────────────────────────────────────┬──────────────────────────────────────────────────┐
 │ 场景            │ 问题                               │ 解决方案                                         │
 ├─────────────────┼────────────────────────────────────┼──────────────────────────────────────────────────┤
 │ 守护进程        │ perf record 默认采样前台进程       │ 必须用 -p $PID                                   │
 ├─────────────────┼────────────────────────────────────┼──────────────────────────────────────────────────┤
 │ 符号被 strip    │ Cargo.toml 默认 strip=true         │ 用 [profile.profiling] 继承 release 但关闭 strip │
 ├─────────────────┼────────────────────────────────────┼──────────────────────────────────────────────────┤
 │ tokio 异步      │ 调用栈可能是 tokio runtime 的 poll │ 看火焰图时跳过 tokio::runtime::* 层              │
 ├─────────────────┼────────────────────────────────────┼──────────────────────────────────────────────────┤
 │ fanotify 内核态 │ perf_event_paranoid 限制           │ sudo sysctl kernel.perf_event_paranoid=-1        │
 ├─────────────────┼────────────────────────────────────┼──────────────────────────────────────────────────┤
 │ 大量短命进程    │ proc connector 事件可能被采样到    │ 关注用户态函数，忽略内核态                       │
 └─────────────────┴────────────────────────────────────┴──────────────────────────────────────────────────┘

 ────────────────────────────────────────────────────────────────────────────────

 这就是以 fsmon 为例的完整 perf 流程。核心思路是：先用 stat 摸底 → 再用 record 采样 → 用 report/flamegraph 找热点 → 用 annotate 定位到具体代码行。
