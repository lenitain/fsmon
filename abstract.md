# filetrace 项目完整提纲

## 一、项目核心定位

### 1. 核心价值

轻量级、高性能的跨平台文件变更溯源工具，实时监控文件操作（创建/删除/修改/大小变化），关联进程ID/进程名/用户，支持快速溯源和筛选，解决"服务器文件变更找不到元凶"的运维痛点。

### 2. 核心差异（对比现有工具）

| 对比项         | inotifywait | auditd | filetrace（本项目） |
|----------------|-------------|--------|---------------------|
| 性能（百万文件监控） | 内存>100MB，卡顿 | 配置复杂、日志臃肿 | 内存<5MB，秒级响应 |
| 进程溯源       | 不支持      | 支持但日志难解析 | 直接关联PID/进程名/用户 |
| 实时性         | 单事件处理  | 延迟高 | 批量事件处理，无延迟 |
| 易用性         | 纯命令行，无筛选 | 需配置规则 | TUI/CLI双界面，多维度筛选 |
| 跨平台         | 仅Linux     | 仅Linux | Linux/macOS（核心），Windows（兼容） |

### 3. 目标用户

运维工程师、服务器管理员、云原生运维、NAS用户。

## 二、核心功能（分阶段实现）

### 阶段1：MVP（最小可用版本，核心代码4000行内）

#### 1. 基础监控能力

- 支持监控指定目录/全盘的文件变更事件：
  - 操作类型：创建（CREATE）、删除（DELETE）、修改（MODIFY）、重命名（RENAME）、大小变化（SIZE_CHANGE）；
  - 触发时机：实时捕获，无漏报（基于Linux inotify/macOS FSEvents/Windows ReadDirectoryChangesW）。
- 事件记录维度：
  - 基础信息：文件路径、操作类型、触发时间（精确到毫秒）、文件大小变化；
  - 核心溯源：进程ID（PID）、进程名（COMMAND）、所属用户（UID/USERNAME）。
- 轻量级存储：
  - 内存缓存最近1小时的变更事件（可配置）；
  - 可选持久化到本地文件（JSON/CSV格式，仅记录元数据，不记录文件内容）。

#### 2. 基础查询能力（CLI）

- 按时间范围筛选：`filetrace query --since "2024-05-01 10:00" --until "2024-05-01 11:00"`；
- 按进程筛选：`filetrace query --pid 1234` 或 `filetrace query --cmd java`；
- 按操作类型筛选：`filetrace query --type MODIFY`；
- 按文件大小变化筛选：`filetrace query --min-size 1GB`（筛选新增/修改超过1GB的文件）。

#### 3. 性能优化（MVP核心）

- 异步事件处理：基于tokio异步运行时，批量处理inotify/FSEvents事件，减少系统调用开销；
- 无锁数据结构：用Rust的`dashmap`存储事件，高并发下无性能瓶颈；
- 内存限制：可配置最大内存占用（默认5MB），超出后自动清理最老事件。

### 阶段2：进阶功能（代码量新增1000-2000行）

#### 1. TUI交互式界面

- 核心视图：
  - 实时事件流（按时间排序，高亮大文件/高危操作）；
  - 进程维度统计（哪个进程修改的文件最多/占用空间最大）；
  - 目录维度统计（哪个目录的变更最频繁）；
- 交互操作：
  - 按方向键导航、按快捷键筛选（如按P筛选进程、按T筛选时间）；
  - 导出筛选结果到文件；
  - 暂停/恢复监控。

#### 2. 增量持久化

- 轻量级数据库（sled）存储历史事件（仅保留7天，可配置）；
- 增量写入：仅追加新事件，不覆盖旧数据，写入性能<1ms/条；
- 快速查询：基于索引按进程/时间/目录查询历史事件，响应<100ms。

#### 3. 跨平台兼容

- 完善macOS（FSEvents）、Windows（ReadDirectoryChangesW）的监控适配；
- 统一CLI/TUI交互逻辑，跨平台行为一致。

### 阶段3：高级功能（代码量新增1000行内，可选）

#### 1. 告警能力

- 自定义告警规则：如"某个进程1分钟内写入超过5GB文件""/etc目录被修改"；
- 告警方式：终端弹窗、邮件、Webhook（支持钉钉/企业微信）。

#### 2. 进程溯源增强

- 关联进程的父进程ID（PPID）和启动命令；
- 记录进程的启动用户和所属组；
- 识别进程是否为容器内进程（Docker/Containerd）。

#### 3. 性能极致优化

- 支持`getdents64`批量系统调用（Linux），进一步降低监控开销；
- 内存映射存储事件日志，减少磁盘IO；
- 支持多线程并行查询。

## 三、技术选型（Rust生态）

### 1. 核心依赖

| 功能领域       | 依赖库                | 选择理由                                  |
|----------------|-----------------------|-------------------------------------------|
| 异步运行时     | tokio                 | 高性能、跨平台，支持异步IO/多线程         |
| 文件监控       | notify-rs             | 跨平台文件变更监控抽象层（封装inotify/FSEvents） |
| 进程信息获取   | procfs（Linux）/sysinfo | 轻量级获取PID/进程名/用户信息             |
| 无锁数据结构   | dashmap               | 高并发下的HashMap，无锁设计               |
| 持久化存储     | sled                  | 轻量级嵌入式KV数据库，无需部署，性能高    |
| TUI界面        | ratatui + crossterm   | 现代、高性能的终端UI库，跨平台            |
| 命令行解析     | clap                  | Rust生态最成熟的CLI解析库，支持自动生成帮助文档 |
| 时间处理       | chrono + humantime    | 精确时间处理，支持人类可读的时间格式      |

### 2. 编译/部署

- 静态编译：生成单文件二进制，无需依赖库，支持Linux/macOS/Windows；
- 交叉编译：支持ARM架构（路由器/NAS）；
- 体积优化：开启`strip`和`opt-level = 3`，二进制体积<5MB。

## 七、用户使用示例

### 1. 基础监控

```bash
# 监控/var/log目录
filetrace monitor /var/log

# 监控全盘，仅记录大小变化超过1GB的文件
filetrace monitor / --min-size 1GB
```

### 2. 查询历史事件

```bash
# 查询过去1小时内修改的文件，按进程筛选java
filetrace query --since 1h --cmd java

# 查询凌晨2点到3点之间，删除的文件
filetrace query --since "2024-05-01 02:00" --until "2024-05-01 03:00" --type DELETE
```

### 3. TUI交互

```bash
# 启动TUI界面，监控/目录
filetrace tui /
```
